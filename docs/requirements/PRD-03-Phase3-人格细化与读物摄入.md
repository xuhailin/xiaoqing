## 一、人格细化与成本优化

### 进化体验优化

* 与「记忆总结」联动：在手动总结或自动总结完成并写入阶段/长期记忆后，会生成进化建议；核心人格变化仍需进化池校验 + 用户确认后写入，用户偏好类变化可直接落地，从而减少用户单独触发更新人格的次数。

### 成本相关约束

* **token 使用必须可预估**：上下文有明确 N 轮 + M 条记忆上限，请求前可估算并截断（见 PRD-00 非功能性要求）。
* 成本优化：在满足体验的前提下，控制注入条数、总结频率与单次 token 用量。

## 二、知识 / 读物摄入（可选模块）

伙伴可「学习」外部读物（如一本书），形成**本地可存的观念或看法**，并可选地参与人格进化。与「掉模型」的区别：模型自带泛化知识；**学一本书 = 把这本书的视角固定到本地**，使回复稳定地体现该书或伙伴对该书的看法。

### 两种学习模式

| | 中性提炼 | 人格化解读（推荐） |
|--|--|--|
| **在做什么** | 把书的观点、论据抽出来，像摘要/读书笔记 | 以**当前人格**为视角，对书做**解读、取舍、保留/存疑** |
| **输出** | 「这本书认为 A、B、C」 | 「以我（思考伙伴、不急于给建议…）的立场，我对 A 认同 / 对 B 会这样 nuance / 对 C 保持距离，我的看法是…」 |
| **存进本地** | 书的「内容要点」 | 伙伴**对这本书的理解与立场**（人格化的看法） |

* **人格化解读**：输入 = 书的文本（或分块）+ **当前人格描述**；prompt 要求 LLM 以当前人格身份输出「对这本书的相对理解与自己的看法」，使同一本书在不同人格下产生不同解读，且与伙伴已有观念一致。

### 流程（与现有设计兼容）

1. **输入**：用户提供一本书的全文或章节（上传 txt/epub 解析为文本，或粘贴）。
2. **处理**：
   - **中性提炼**：LLM 提取核心观点、立场、金句 → 候选「书的内容要点」。
   - **人格化解读**：LLM 在人格 + 进化池约束下，输出「伙伴对这本书的看法」→ 候选「观念/看法」。
3. **存储**：候选先不入人格层，存入「读物学习结果」或待采纳列表（可单独表或结构），与记忆/人格区分。
4. **选择是否进化**：
   - 用户在前端查看候选，勾选「采纳哪几条进入人格/长期记忆」后确认；和/或
   - LLM 根据当前人格 + 进化池 + 候选，输出「建议采纳 A、B，不采纳 C」→ 再经**进化池校验 + 用户确认**后写入。
5. **写入**：通过校验且确认的条目写入人格层或长期记忆，之后对话时按现有机制注入，伙伴即「带着对这本书的看法」回复。

### 实现与阶段建议

* 可放在 **Phase 3** 作为「知识/读物摄入」模块；V1 可不实现，或注明「读物摄入为后续阶段」。
* 前端：支持上传/粘贴文本、选择「中性提炼」或「人格化解读」、查看候选与确认采纳。
