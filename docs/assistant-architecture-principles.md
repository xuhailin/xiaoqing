# 小晴助手架构原则

> 本文档定义小晴对话/助手管线的长期架构目标与设计规则。  
> 修改或扩展 `backend/src/xiaoqing/**` 及相关管线时，必须遵循本原则。

---

## 1. 长期目标

小晴助手应表现为**一个连贯的智能体**。

其核心能力是：

- **高质量对话**
- **正确的行动决策**

架构必须支撑：

- 清晰的推理链路
- 稳定的决策机制
- 一致的人设与表达
- 可长期维护与演进

---

## 2. 核心架构原则：五能力管线

助手管线必须围绕 **5 个能力** 组织：

| 能力 | 职责 |
|------|------|
| **1. 感知（Perception）** | 理解用户输入、情境、上下文，产出结构化认知结果，不做行为决策。 |
| **2. 决策（Decision）** | **唯一**做出「本回合要执行什么行动」的层级；集中决策权，不分散。 |
| **3. 执行（Execution）** | 根据决策结果调用工具、外部服务、内部能力，并汇总执行结果。 |
| **4. 回复组织（Response Composition）** | 根据决策与执行结果，组织最终回复内容与形式（表达层）。 |
| **5. 回合后处理（Post-Turn Processing）** | 记忆写入、总结、成长、埋点等回合结束后的收尾工作。 |

每个能力有**清晰且单一**的职责，**不应出现多层做类似决策**的情况。

---

## 3. 设计规则

### 3.1 决策权集中

- **只有一层**负责做出「行动决策」。
- 不得将决策逻辑分散在：意图识别、推理层、行动规划器、Prompt 路由等多处。
- **决策权必须集中**在决策层（Decision）。

### 3.2 意图不得成为隐式决策引擎

- 意图识别应**描述用户含义**（用户想表达什么、属于哪类需求）。
- **最终采取什么行动**必须由决策层决定，而不是由意图层「顺带」决定行为。

### 3.3 回复生成不得再次决定系统行为

- 对话模型应专注于**表达**（怎么说、写什么内容）。
- 不应在生成回复时再次做系统路由或行为选择，避免「生成即决策」的耦合。

### 3.4 避免重复的推理层

- 若两个组件都在「分析用户含义」或「决定要做什么」，必须合并或明确划分职责。
- 避免意图、推理、策略等多处重复解读用户，导致不一致与难维护。

### 3.5 避免增量打补丁

- 当某次改动影响助手管线时，应先考虑**是否应简化结构**，而不是在现有结构上继续打补丁。
- 优先减少层级与分支，再考虑局部修改。

---

## 4. 重构哲学

因小晴**尚未正式上线**：

- **允许结构性重构**：可以调整管线阶段、合并或拆分模块。
- **可移除过时层级**：若某层已与上述五能力冲突或重复，可下线而非保留兼容。
- **不强制向后兼容**：在架构清晰的前提下，可以打破旧接口或旧行为。

优先保证：

- 架构清晰
- 推理一致
- 长期可维护

---

## 5. 新增能力时的自检

在添加新功能或新模块时，必须回答：

**该能力属于哪一层？**

- 感知（Perception）
- 决策（Decision）
- 执行（Execution）
- 回复组织（Response Composition）
- 回合后处理（Post-Turn Processing）

若**无法明确归属到单一层**，则设计需要重新考虑（拆分职责或合并到已有层）。

---

## 6. 总体目标

小晴应演进为**清晰的认知管线**，而不是零散逻辑的堆砌。

**架构清晰优先于临时兼容。**  
本文档与 `CLAUDE.md`、`.cursor/rules` 中的助手架构约定一致，作为后续开发与重构的基准。
