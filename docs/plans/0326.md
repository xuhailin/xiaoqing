# 任务背景

当前项目是小晴系统，我现在想支持一个“多用户模式”。

目标不是立刻做完整 SaaS，而是先支持：
- 系统可切换为“单用户模式 / 多用户模式”
- 多用户模式下，增加一份独立配置（可以从现有 env / 配置复制演化）
- 先只保留核心 chat / 基础会话能力
- 以下能力在多用户模式下暂时关闭或禁止暴露：
  - devagent
  - designagent
  - openclaw
  - 本地技能下载
  - 打工相关能力
  - 工时上报相关能力

我记得之前系统里“好像默认就有一些多用户痕迹”，但不确定那是真支持，还是只是表面上留了字段 / sessionKey / conversationId 之类。
所以这次你的任务是：**基于代码事实，审查这个方案是否可行，以及需要补哪些关键边界。**

# 审查目标

请你围绕下面 6 个问题展开审查：

## 1）当前系统到底是“单用户系统”还是“伪多用户 / 半多用户系统”
请基于代码事实判断：
- 用户身份是否真的存在
- 会话、记忆、关系、世界状态、计划、提醒、任务、认知状态等数据，是否带 userId / tenant / owner 边界
- 哪些模块已经具备多用户基础
- 哪些模块其实默认写死为单用户

不要只看 schema，也要看真实查询和写入逻辑。

---

## 2）如果引入“多用户模式配置开关”，是否能以最小侵入方式落地
我希望有一个类似：
- SINGLE_USER_MODE=true/false
或
- APP_USER_MODE=single|multi

并且多用户模式可以配一份独立配置文件（可以复制现有 env 演化）。

请你审查：
- 当前配置体系是否适合引入这个开关
- 配置读取是否集中
- 哪些模块会因为这个开关发生分支
- 是否适合通过 Feature Flag / Capability Gate / Module Registration 的方式关闭能力
- 是否需要新增单独的 multi-user env / config 文件

请给出推荐方案，而不是只说“可以”。

---

## 3）多用户模式下，禁用指定能力是否足够安全
我要暂时关闭：
- devagent
- designagent
- openclaw
- 本地技能下载
- 打工相关能力
- 工时上报

请审查：
- 这些能力当前是如何注册、暴露、路由、执行的
- 关闭它们，应该关在哪一层最稳妥：
  - 前端入口隐藏
  - capability registry 不注册
  - intent router 不路由
  - 后端执行层拒绝
  - prompt / skill 可见性控制
- 哪些地方如果只关 UI 还会留下后门
- 哪些能力与 chat 主链路耦合过深，不能简单关闭

输出时请给出“推荐关闭层级”。

---

## 4）多用户模式下最容易发生串数据的问题在哪里
重点审查这些方面是否会串：
- conversation / session
- memory / claim / relationship / persona / identity anchor
- reminder / plan / task
- world state / daily moment / summarizer / post-turn
- 本地缓存 / 进程内状态 / singleton service
- 文件下载目录 / 工作目录 / 临时文件
- websocket / SSE / 前端 store
- dev-agent workspaceRoot / run state（即便多用户先关闭，也要看遗留风险）

请按“高风险 / 中风险 / 低风险”分类。

---

## 5）多用户模式的最小可行落地路径是什么
注意：我现在要的是“先可用，不是一次到位”。

请基于代码事实，给我一个最小落地方案，例如：
- Phase A：只是配置开关 + 能力禁用 + UI 收口
- Phase B：补 owner/userId 边界
- Phase C：清理进程内单例态 / 本地文件隔离
- Phase D：再考虑真正的多用户登录与权限

你需要自己根据代码现状判断最合理的阶段划分，不要机械套模板。

---

## 6）是否值得做，还是应该先明确为“单用户系统”
这是很重要的一问。
请你最后明确判断：
- 当前代码基础上，这个“多用户轻量模式”是不是值得现在做
- 它是“低成本可落地”
- 还是“表面可做，实际牵一发动全身”
- 还是“应该先把系统明确收口为单用户，再考虑多用户”

请给出鲜明结论，不要模糊表述。
