# Think · 深度架构思考

在任何实现动作之前，先**想清楚**。本 skill 的产出是「供用户做取舍决策的思考报告」，而不是实现方案。

---

## 适用场景

- 用户提出一个新功能/需求，但尚未明确实现路径
- 用户对某一改动有疑虑，想先评估再动手
- 用户说"先想一想""think 一下""分析一下""有没有坑"等
- 任何改动涉及核心边界（执行安全、记忆存储、路由策略、数据模型）

---

## 模式判断（必须先执行）

根据问题复杂度选择输出模式：

### 轻量模式（默认）

适用于：
- 用户只是想快速判断方向（如「这个要不要改」「这样设计行不行」）
- 需求范围较小（单模块 / 单页面 / 单链路）
- 不涉及核心边界（执行安全 / 调度 / 记忆 / 数据模型）

输出格式（固定结构）：

```
## 快速方向判断

**目标理解**：（一句话复述用户想做什么）

**当前现状**：（相关模块/逻辑现在是什么状态，1~2句）

**关键风险**（1~3条）
- ...

**建议方向**
- 是否值得做：是 / 否 / 有条件（说明原因）
- 是否进入 plan：是 / 否（若否，说明原因）
```

### 完整模式（深度分析）

适用于：
- 涉及核心能力或边界（执行链路、调度、记忆、数据模型）
- 可能引入结构调整或新模块
- 用户明确要求「深度分析」「全面评估」「长期看怎么做」

输出要求：
- 使用完整思考框架（目标 / 现状 / 方案对比 / 长期架构 / 风险 / 决策点）
- 明确方案取舍和长期影响
- 若信息不足，列出「需要确认的假设」，但**基于当前信息先给出初步判断**，避免中断分析流程

---

## 思考框架（完整模式专用）

### 1. 厘清目标层次

从两个层次明确「要做什么」：

- **系统目标**：这个改动在小晴整体架构中解决什么问题？与哪个核心能力相关（感知 / 决策 / 执行 / 回复组织 / 回合后处理）？
- **功能目标**：用户具体想要的行为是什么？最小可验证的完成标准是什么？

不清楚时，先列出「假设的目标」并标注「需用户确认」。

---

### 2. 现状摸底

改动前，先快速描述现实：

- 当前相关模块/服务是什么？入口在哪？
- 现有数据流是什么路径？
- 已有哪些约束或边界（来自 `CLAUDE.md`、架构文档、安全策略）？

**不猜测，先读代码**（用 Glob / Read / Grep 确认真实路径）。

---

### 3. 方案选项

列出 **2 ~ 3 个可能的实现方向**，每个方向包含：

| 方向 | 核心思路 | 改动范围 | 是否符合现有架构边界 |
|------|----------|----------|----------------------|
| 方案 A | ... | 小 / 中 / 大 | 是 / 否，原因 |
| 方案 B | ... | 小 / 中 / 大 | 是 / 否，原因 |

---

### 4. 长期架构视角

站在 **6 个月后**看这个决策：

- 哪个方案在扩展时最不容易变成债？
- 是否存在「现在省事但将来必须重写」的设计？
- 新能力是否能明确归属到现有模块职责，还是需要引入新模块？
- 对记忆、人格、DevAgent 隔离边界有无副作用？

---

### 5. 风险点与隐患

至少输出 **3 个风险点**，按严重程度排序：

- **高风险**：可能导致数据丢失、执行错误、安全边界被突破
- **中风险**：可能导致逻辑错误、UI 状态不一致、性能劣化
- **低风险**：可能引入技术债或维护困难，短期无感知

每条风险要具体到**模块 / 文件 / 数据流**，不泛泛而谈。

---

### 6. 建议与决策点

给出你的**倾向方案 + 核心理由**，然后列出需要用户拍板的决策点：

- 决策点 A：xxx（选择影响 xxx）
- 决策点 B：xxx（选择影响 xxx）

---

## 输出格式

```
## 目标确认
- 系统目标：...
- 功能目标：...
- 待确认假设：（如有）...

## 现状摘要
- 相关模块：...
- 当前数据流：...
- 关键约束：...

## 方案对比
| 方向 | 核心思路 | 改动范围 | 架构合规 |
|------|----------|----------|----------|
| A    | ...      | 小       | 是       |
| B    | ...      | 中       | 部分     |

## 长期架构评估
...

## 风险清单
1. [高] ...
2. [中] ...
3. [低] ...

## 建议与决策点
- 倾向：方案 X，因为 ...
- 决策点 1：...
- 决策点 2：...
```

---

## 原则与禁止事项

- **禁止在 Think 阶段直接改代码或输出完整实现方案**。
- 若用户未明确说"开始做"或"按这个方案走"，不要自动进入实现。
- 不美化、不回避问题；有风险直接说，不因是个人项目降低标准。
- 若信息不足，列出「需要确认的假设」，但基于当前信息先给出初步判断，避免中断分析流程；完整模式下可追问关键未知项。
