# Skills 使用指南

三个核心 skill 构成「想 → 计划 → 核查」工作流，支持从快速小改到深度架构分析的全场景覆盖。

---

## 快速上手：说什么触发哪个

### think — 动手前先想清楚

| 你说的话 | 触发模式 |
|----------|----------|
| 「这样设计行不行？」「要不要改这里？」 | 轻量 |
| 「有没有坑？」「先想一想」「think 一下」 | 轻量 |
| 「深度分析一下」「长期看怎么做」「全面评估」 | 完整 |
| 涉及调度 / 执行安全 / 记忆 / 数据模型 | 完整 |

**轻量输出**：目标理解 + 现状 + 1~3 个关键风险 + 建议方向（要不要做 / 要不要进 plan）

**完整输出**：目标 / 现状 / 2~3 方案对比 / 长期架构评估 / 风险清单 / 决策点

---

### plan — 出一份可执行的计划

| 你说的话 | 触发模式 |
|----------|----------|
| 「帮我列一下步骤」「怎么改？」「直接出计划」 | 轻量（改动小时）|
| 「出完整计划」「plan 一下」「逐任务拆解」 | 完整 |
| 涉及多文件 / Schema 变更 / 新模块 | 完整 |

**轻量输出**（1~2 文件小改动）：目标 + 改动文件 + 验收条件 + 风险

**完整输出**：背景 / 验收标准（AC）/ 逐 Task 拆解 / 执行顺序 / 执行策略 / 验收步骤

---

### review — 检查改的对不对

| 你说的话 | 触发模式 |
|----------|----------|
| 「帮我看看对不对」「有没有问题」「看一眼」 | 轻量 |
| 「review 一下」「验收一下」「逐条检查」 | 完整 |
| 有明确 plan + 多文件改动 | 完整 |

**轻量输出**：是否有明显问题 / 可能遗漏的情况 / 风险点 / 建议（最多3条）+ 「可以先用」结论

**完整输出**：AC 逐条对照（PASS/FAIL/PARTIAL）/ 架构合规 / BLOCKER / 建议改进

## 典型日常场景

**场景 A：小 bug / 快速修一行**
```
你：「帮我看一下这个 return {} 要不要改」
→ think 轻量：确认风险，建议改
→ plan 轻量：直接出改动 + 验收
→ 实现
→ review 轻量：「可以先用」
```

**场景 B：中等功能（3~5 文件）**
```
你：「plan 一下怎么加 conversationId 校验」
→ plan 完整：AC + 逐 Task + 执行策略
→ 实现
→ 你：「review 一下」→ review 完整
```

**场景 C：不确定要不要做**
```
你：「think 一下，这个提醒功能要不要加 session 绑定」
→ think 轻量：给建议方向
→ 你确认后 → plan → 实现 → review
```

**场景 D：架构级改动**
```
你：「深度分析一下调度模块怎么重构」
→ think 完整：方案对比 + 长期评估 + 决策点
→ 你拍板 → plan 完整 → 实现 → review 完整
```

### design-audit — 前端设计系统一致性审查

| 你说的话 | 触发模式 |
|----------|----------|
| 「审查 memory 页面」「设计 audit」「帮我看一下这个页面的 UI 规范」 | 代码审查 |
| 上传截图 + 说明 | 视觉审查 |
| 同时指定页面路由和截图 | 代码 + 视觉全量审查 |

**输出**：结构化 `audit_result` JSON + 可读汇总，包含：总体状态 / findings（规则违规）/ 最小修改计划 / 不可动区域

与后端 `DesignAgentService` 等价，不需要服务运行，直接通过工具读取代码和知识库完成审查。支持后续多轮：描述问题、请求修改方案、确认执行修改。
