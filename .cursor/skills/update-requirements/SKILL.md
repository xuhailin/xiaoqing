# 更新需求与文档同步（LongMemory 项目）

## 使用时机

当用户说「更新需求」「改一下这个需求」「这个功能的行为要调整一下」等与**需求 / 行为变更**相关的话题时，使用本 skill。

目标：在实现或修改功能时，**同时把 docs 里的需求说明更新到最新状态**，并且覆盖掉已经过期的描述。

## 操作步骤

1. **识别变更范围**
   - 分析用户这次「更新需求」影响的是哪一层、哪些模块：
     - 仅后端行为？仅前端交互？还是 Prompt / 人格 / 记忆注入策略？
     - 是否涉及数据库模型（Prisma）、API 结构、或整体能力边界？

2. **映射到对应文档**
   - 按下面规则决定要改哪个 / 哪些 docs（可多选）：
   - **全局能力 / 模块结构 / API 总览 / 数据库结构**
     - 改：`docs/PROJECT-SUMMARY.md`
   - **总体约定 / 禁止事项 / 阶段目标 / 术语**
     - 改：`docs/requirements/PRD-00-总览与约定.md`
   - **Phase 1：基础对话与记忆**
     - 改：`docs/requirements/PRD-01-Phase1-基础对话与记忆.md`
   - **Phase 2：记忆注入与人格双池（Persona + 注入规则 + Prompt Router 相关）**
     - 改：`docs/requirements/PRD-02-Phase2-记忆注入与人格双池.md`
   - **Phase 3：人格细化与读物摄入（Reading / ReadingInsight + 进化策略 + Token 估算等）**
     - 改：`docs/requirements/PRD-03-Phase3-人格细化与读物摄入.md`

3. **更新原则**
   - 以「当前代码行为 + 用户口头最新约定」为准，**允许直接覆盖旧描述**，不需要保留详细历史版本。
   - 保持 `PROJECT-SUMMARY.md` 与各 PRD 描述一致，如有出入要一起修。
   - 不把「以后要实现」和「已经实现」混在一起，必要时拆成「已实现能力」和「计划中」两个小节。

4. **具体执行（对每个受影响文档）**
   1. 用工具读取对应文档。
   2. 找到与这次变更相关的小节（例如某个 API、小流程、Persona 字段说明等）。
   3. 按用户最新需求改写为**当前唯一真实版本**：
      - 删除或改写已过期的约束 / 字段说明 / 流程描述。
      - 如有新字段、新步骤，补充到表格或列表中。
   4. 简要自查：
      - 文档是否能独立被人读懂，不依赖这次聊天上下文？
      - 是否准确反映当前实现，而不是「理想状态」？

5. **向用户汇报**
   - 用极简总结说明：
     - **本次需求变更点**
     - **改了哪些 docs 文件的哪些部分（概念级描述即可）**
   - 不需要在总结里贴完整文档内容，除非用户特别要求。

## 使用示例（对话层面）

- 用户：「这个总结接口现在不要自动写长期记忆了，只给候选就行，帮我更新一下需求。」
  - 识别：涉及总结模块 + 记忆写入策略（Phase1/2 边界），禁止自动长期记忆。
  - 文档：
    - 更新 `PRD-01` 或 `PRD-02` 中关于总结和记忆写入的规则。
    - 若影响整体「禁止事项」，同步改 `PRD-00`。

- 用户：「Persona 里约束池多加一个字段，前端配置页也要跟着改，帮我把需求也改一下。」
  - 识别：Persona 模型 + 前端人格配置交互 → Phase2。
  - 文档：
    - 更新 `PRD-02` 里 Persona 模型表和交互说明。
    - 若数据库模型有变更，同时检查 `PROJECT-SUMMARY.md` 的模型列表是否要改。
