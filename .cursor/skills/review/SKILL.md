# Review · 实现核查

本 skill 以「计划 → 代码对照」为核心工作流，产出明确的**通过 / 部分通过 / 不通过**判定，以及具体的改进建议。

---

## 前置条件

- **若存在 plan**：优先按 plan 的验收标准逐条对照核查。
- **若没有 plan**：基于代码改动自动推断目标（从 diff / 文件改动 / 用户描述中提取），再按推断目标进行审查，并在输出开头说明推断的目标是什么。

无论哪种情况，**必须基于实际代码内容进行判断，不得凭猜测**。优先通过 Glob / Grep / Read 搜索、定位、阅读代码确认逻辑是否存在。

---

## 模式判断（必须先执行）

在开始核查前，根据用户输入和改动复杂度选择审查模式：

### 轻量模式（默认）

适用于：
- 用户只说「帮我看看对不对」「有没有问题」之类的模糊请求
- 改动范围小（单文件 / 少量逻辑）
- 没有明确 plan

输出格式（固定结构）：

```
## 快速检查结论

### 是否有明显问题
- （有则描述，无则写「无」）

### 可能遗漏的情况
- （边界 / 异常路径 / 未处理的输入，无则写「无」）

### 风险点（如果有）
- （具体到文件 / 逻辑，无则省略本节）

### 建议（最多 3 条）
- （非阻塞，可选跟进）

👉 若当前实现已经基本可用，明确说明「可以先用」
```

### 完整模式（严格审查）

适用于：
- 用户提供了 plan 并要求对照验收
- 明确说「review」「验收一下」「逐条检查」
- 涉及多文件 / 核心执行链路 / 关键功能（如 scheduler、路由、安全策略）

输出要求：
- 使用完整结构（AC 逐条对照 / PASS / FAIL / PARTIAL 表格）
- 覆盖架构合规检查与风险分析
- BLOCKER 必须给出文件行号 + 修改建议

---

## 核查流程（完整模式专用）

> 轻量模式直接使用「模式判断 → 轻量模式」中的固定输出格式，不走以下步骤。

### Step 1. 获取改动事实

**必须先读代码，不凭记忆或猜测。**

- 用 `Glob` 找到改动涉及的文件
- 用 `Read` 读取实际内容
- 用 `Grep` 确认关键逻辑是否存在（如校验分支、错误抛出点、字段传递）

---

### Step 2. 对照计划逐条核查

对计划中的每个 Task / 验收标准，独立判定：

| 编号 | 验收条件 | 状态 | 实际代码位置 | 问题描述（如有） |
|------|----------|------|--------------|------------------|
| AC-1 | ... | PASS / FAIL / PARTIAL | `file:line` | ... |
| Task 1 | ... | PASS / FAIL / PARTIAL | `file:line` | ... |

状态说明：
- **PASS**：代码行为与预期完全一致
- **PARTIAL**：核心逻辑正确，但存在遗漏或边界未处理
- **FAIL**：代码与预期不符，或根本未改动

---

### Step 3. 优先核查三个核心问题

在进入架构规范和代码质量之前，**先回答这三个问题**：

1. **是否达成功能目标**：改动后，用户描述的功能是否可以工作？
2. **执行链路是否闭环**：从入口到结果，整条链路是否都有对应实现（无断点、无静默失败）？
3. **用户是否可感知**：成功或失败的结果，用户能否在 UI / 日志 / 消息中感知到？

若三项均满足且无关键风险，**明确说明「可以先使用，无需继续优化」**，不强行输出建议改进列表。

---

### Step 4. 架构合规性检查（三项均满足后才进行）

除功能正确性外，还要核查以下维度：

**边界与职责**
- 改动是否跨越了不该跨越的模块边界（如 DevAgent 调用了 Memory、chat 主链被注入了 dev 逻辑）？
- 新增的服务/方法是否归属于正确的模块？

**安全性**
- 若涉及 Shell 执行：是否沿用了现有白名单 / 黑名单策略？
- 若涉及路径操作：是否有路径校验，防止目录穿越？
- 若涉及用户输入：是否有必要的边界校验？

**错误处理**
- 失败路径是否真实抛错（而非 warn + return {}）？
- 错误是否能被上层捕获并正确记录？

**数据一致性**
- 若新增字段：DTO / Prisma schema / 前端类型是否同步更新？
- 若改了枚举/常量：所有引用方是否都已对齐？

**Feature Flag 合规**
- 若新增 feature flag：是否同步更新了 `.env.example`（遵守 `CLAUDE.md §5`）？

---

### Step 5. 代码质量检查（次要，非阻塞）

针对改动范围内的代码，检查以下问题（**不扩展到未改动的代码**）：

- 是否存在过度工程化（为当前需求新增了不必要的抽象）？
- 是否有魔法数字 / 硬编码字符串（应改为常量或配置）？
- 前端改动是否使用了 design token，而非硬编码颜色 / 间距（遵守 `ui-design-system.mdc`）？
- 是否有明显的 N+1 查询或不必要的数据库操作？

---

### Step 6. 综合判定与输出

**改动较简单时（单文件 / 逻辑清晰）**，用列表代替表格，避免冗余输出：

```
## Review 结论

**总体判定**：PASS（可以先使用，无需继续优化）

**核心目标核查**
- [PASS] 功能目标：createPlan 在 conversationId 缺失时抛 400，`plan.service.ts:89` 已实现
- [PASS] 执行链路闭环：入口 → 校验 → 抛错 → 上层捕获，链路完整
- [PASS] 用户可感知：前端会收到 400 响应，控制台有错误日志

**已确认正确**
- Task 1：`backend/src/plan/plan.service.ts:89` conversationId 校验逻辑正确
```

**改动较复杂时（多文件 / 有依赖链）**，用表格展示：

```
## Review 结论

**总体判定**：PARTIAL / FAIL

### 核心目标核查
| 项目 | 状态 | 说明 |
|------|------|------|
| 功能目标 | PASS | ... |
| 执行链路闭环 | FAIL | notify-dispatch:45 仍 return {}，链路不闭合 |
| 用户可感知 | PARTIAL | 成功可感知，失败无日志 |

### 验收标准核查
| AC | 条件 | 状态 | 说明 |
|----|------|------|------|
| 1  | ...  | PASS | `file:line` |
| 2  | ...  | FAIL | 未找到对应改动 |

### 必须修复（BLOCKER）
1. `path/to/file.ts:line` — 问题描述 → 建议修改方式

### 建议改进（非阻塞，最多 3 条）
1. `path/to/file.ts:line` — 问题描述 → 建议

### 已确认正确
- Task 1：`backend/src/plan/plan.service.ts:89` conversationId 校验逻辑正确
```

---

## 输出原则

- **只评价改动范围内的代码**，不扩展到无关模块（避免 scope creep）。
- **BLOCKER 必须给出具体文件行号 + 建议修改方式**，不能只说"有问题"。
- **非阻塞建议最多 3 条**，不要把所有代码风格问题都列出来。
- **PASS 的项也要列出**，让用户看到哪些已经确认正确，减少焦虑。
- **不替用户做决定**：PARTIAL 或 FAIL 时，说明问题并给建议，由用户决定是否修复。

---

## 禁止事项

- **禁止在 Review 阶段直接修改代码**，除非用户明确说「你帮我改」。
- **不凭直觉判定 PASS**，必须读代码确认。
- **不扩大审查范围**，用户让你 review Task 1，不要顺便把整个模块都 review 了。
- **不新增测试文件**（遵守 `no-new-tests-by-default.mdc`），即使发现测试覆盖不足。
