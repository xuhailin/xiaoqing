# Xiaoqing Architecture Health Check

请对当前仓库做一次 **架构健康检查**。

目标不是新增功能，也不是直接改代码，  
而是检查当前系统是否仍然保持稳定、清晰、边界正确。

请基于当前代码，重点检查以下内容：

## 1. 主链路是否清晰
检查当前 Assistant 主链路是否仍然成立：

User Input
→ Intent
→ Reasoning
→ Route Decision
→ (Chat / Tool / DevAgent / Planning)
→ Response
→ Reflection
→ Memory / DevExperience Writeback

请指出：
- 当前真实链路是什么
- 哪些节点已接入
- 哪些节点缺失
- 是否存在重复判断、链路绕路、职责重叠

---

## 2. Reasoning 是否仍然有效
检查：
- reasoning 是否真正参与路由判断
- 是否只是 prompt 文本存在
- 是否能判断：聊天 / 工具 / dev / planning / 风险操作

如果 reasoning 没有真正生效，请指出。

---

## 3. System Self 是否仍然稳定
检查：
- system self 是否仍然被主链路使用
- 是否只是文本描述
- 是否能稳定影响风格、边界、执行策略
- 是否存在“定义了 system self，但运行时没消费”的问题

---

## 4. Planning 层是否边界清晰、职责对齐

当前存在三处 Planning 相关实现，需分别检查：

**Assistant `TaskPlannerService`**（`assistant/planning/`）：
- `taskPlan` 是否只用于 prompt 文本注入（经 `PromptRouterService` 的 `[任务规划]` 段），而非真正驱动执行步骤
- 产出的通用步骤（”理解需求 → 执行任务 → 确认完成”）是否有实际编排价值
- 是否存在”生成了 plan，但下游没按 plan 执行”的断路

**DevAgent `DevTaskPlanner`**（`dev-agent/planning/`）：
- 任务拆解是否仍在 dev 通路内闭环，不反向影响 Assistant 决策
- `DevReplanPolicy` 是否存在无限重规划风险

**`CapabilityChainExecutor`**（`action/`）：
- 是否已有调用方接入，还是仅为预留架构

---

## 5. Reflection 是否存在且有效
检查：
- 是否有 post-turn reflection
- 是否能发现路由错误、人格漂移、遗漏机会
- 是否只是日志，没有反馈价值
- 是否真正参与 memory / devExperience 写回判断

---

## 6. Memory 边界是否正确
检查是否区分：

Assistant Memory：
- 用户偏好
- 长期目标
- 关系与人格一致性信息

Dev Experience：
- 开发经验
- repo 特性
- bug 处理模式

重点检查：
- DevAgent 是否可能直接写 Assistant Memory
- 是否存在记忆污染
- 是否存在职责越界

---

## 7. Assistant / DevAgent 边界是否清晰
检查：
- Assistant 是否负责主体认知、人格、用户关系
- DevAgent 是否只负责任务执行、开发决策、执行经验
- 是否存在 DevAgent 反向污染 Assistant 的情况
- 是否存在职责重复

---

## 8. Capability / Skill / Tool 边界是否清晰
检查：
- 是否仍然能区分“能力注册”“任务编排”“执行工具”
- 是否存在 Assistant 直接耦合底层执行细节
- 是否存在 DevAgent 越过边界直接操作不该操作的模块
- 是否有能力命名混乱、职责重叠的问题

---

## 9. 通道层（Channel）是否边界清晰
检查：
- Gateway → MessageRouter 的路由逻辑是否稳定
- 各通道（企业微信、Web 等）是否走统一入口
- 是否存在某个通道绕过 Gateway 直接调用内部服务
- 通道特定逻辑（如企业微信加解密）是否正确隔离在 channel 模块内

---

## 10. DevAgent 执行安全是否完整
检查：
- ShellExecutor 的白名单 / 黑名单是否生效
- Workspace 路径校验是否完整（`DEV_AGENT_ALLOWED_WORKSPACE_ROOTS`）
- 是否存在命令注入风险
- worktree 模式的隔离是否正确

---

## 11. 是否出现人格漂移或系统漂移
请从代码结构和 prompt 组织角度判断：

- Assistant 是否越来越像普通 LLM 问答
- 是否丢失“小晴”的身份稳定性
- 是否出现 chat / dev / tool 混成一层的问题
- 是否存在新增功能后把原有认知结构冲散的问题

---

## 12. 输出要求

请只输出四部分：

### A. 当前健康结论
用简短结论说明：健康 / 部分健康 / 存在明显风险

### B. 当前真实运行链路
写成：
User → ... → ... → ...

### C. 主要问题（最多 5 条）
只写最关键的问题，不要发散

### D. 修复建议（最多 5 条）
只给最小修复建议，不要推翻现有架构，不要重新设计整套系统

---

## 约束
不要：
- 直接写代码
- 大规模重构建议
- 发明太多新概念
- 脱离当前代码另起一套架构

只做：
**基于现有代码的健康检查 + 最小修复建议**
