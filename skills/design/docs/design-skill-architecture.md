# Design Skill Architecture

## 1. 设计目标

`Design Skill v1` 的存在，不是为了把小晴变成一个“会自动画 UI 的通用设计平台”，而是为了给小晴当前前端系统建立一层稳定、可复用、能被 AI 明确执行的设计约束能力。

它解决的是当前几个很具体的问题：

- 页面局部优化后，整体风格越来越不统一
- AI 在改 UI 时容易用“卡片、渐变、阴影、玻璃感”补视觉，导致页面越来越重
- 现有 light / dark 已经存在，但缺少一套把主题、页面类型、结构层级连接起来的稳定约束
- 小晴已经有聊天、工作台、记忆、配置等多种页面，未来还会继续扩展，不能每次都从“审美讨论”开始

因此，这个 skill 的目标不是“生成更多 UI”，而是：

- 把小晴已经形成的设计语言固化下来
- 让 AI 在修改页面时先受约束，再做表达
- 让新页面优先继承现有系统，而不是发明新的视觉分支
- 为未来抽象出独立 `design-agent` 做准备

## 2. 设计原则

### 高约束

这个 skill 的核心不是开放创作，而是给 AI 一条窄但稳定的路径。它强调：

- 先判定页面类型，再选择 preset，再落到 token 与 primitive
- 先检查规则冲突，再讨论视觉优化
- 先做最小修改，再考虑结构升级

也就是说，AI 不应该从“我能设计什么”开始，而应该从“这里不允许我做什么”开始。

### 一致性优先

小晴是一个长期演进中的个人 AI 产品。用户会反复使用同一套界面，因此一致性比局部惊艳更重要。这个 skill 默认把下面几件事排在前面：

- 页面之间能否共享同一套层级逻辑
- light / dark 是否仍然是同一产品，而不是两套皮肤
- 控件、卡片、标题、密度是否能被预测
- AI 改完之后，是否让后续页面更容易继续沿用

### 最小修改

`Design Skill v1` 默认不鼓励大改。除非页面已经出现明显结构错误，否则优先通过以下方式解决问题：

- 删除视觉噪音
- 收敛间距、边框、卡片层级
- 把局部样式拉回已有 token / primitive
- 复用已存在的 page shell、section、panel 关系

这意味着它更像“设计约束器”和“设计审查器”，而不是“视觉创意器”。

## 3. 能力边界

### 不做什么

`Design Skill v1` 明确不做以下事情：

- 不做通用 UI 生成平台
- 不做任意风格切换
- 不做完全自由的视觉提案
- 不试图定义一套重量级 DSL 或独立设计语言编译器
- 不把每个页面都做成 hero 化、营销化、截图导向的作品

### 重点做什么

它重点做三类能力：

- `约束`：通过 UI rules、theme tokens、preset 把边界写清楚
- `审查`：通过 review prompt 先识别不一致、层级问题和过度设计风险
- `稳定生成`：在明确页面类型与 preset 的前提下，为新页面提供可控生成模板

换句话说，它不是“从零设计”，而是“在小晴已有系统内稳定地产生正确结果”。

## 4. 与小晴的关系

这个 skill 是小晴当前 UI 系统的一部分，而不是外部工具。

它与小晴的关系有三层：

### 作为 skill 被调用

当 AI 需要审查、优化或生成小晴前端页面时，先调用 `Design Skill v1`，再进入实际代码修改。这样设计判断不会散落在每次对话里，而是复用同一套约束源。

### 服务当前 UI 系统

它不是独立审美体系，而是建立在当前项目事实上已经存在的基础之上：

- `frontend/src/styles/_variables.scss`
- `frontend/src/styles/_design-system.scss`
- `docs/frontend-ui-rules.md`
- chat / workbench / memory 等页面当前已有的风格差异

因此，这个 skill 的价值不是“另起炉灶”，而是把这些已有基础变成 AI 可执行、可复用的能力包。

### 作为未来 design-agent 的前置层

如果未来直接做 agent，而没有先把规则、preset、prompt、theme token 这些基础层收敛清楚，agent 很容易退化成“会说设计意见的自由发挥器”。skill 先把约束层沉淀下来，可以降低未来 agent 化时的漂移风险。

## 5. 未来如何演进为 design-agent

这是 `Design Skill v1` 最重要的长期价值。

### 哪些模块可以拆出去

未来最适合独立出去的模块有四类：

1. `规则模块`
   也就是现在的 `rules/`。未来可以抽象成 agent 的审查规则库，用于自动识别卡片滥用、层级混乱、主题漂移等问题。

2. `主题与 token 模块`
   也就是现在的 `tokens/`。未来可作为统一 theme registry，让 agent 不只是“看起来像”，而是知道该调用哪类语义 token。

3. `preset 选择模块`
   也就是现在的 `presets/`。未来可让 agent 在任务开始时自动判定页面属于哪种风格基线，并只做受控偏移。

4. `任务模板模块`
   也就是现在的 `prompts/`。未来能演化为 agent 的任务协议层，明确区分 `review / refine / generate` 三种工作模式。

### 如何从 skill 走向 agent

建议路径不是一次性跳到独立 agent，而是分层推进：

第一步，先让 skill 成为“设计前置检查层”。
也就是 AI 在改 UI 前，先经过本 skill 读取规则、主题与 preset，再输出修改建议或方案。

第二步，把 skill 的输出结构稳定下来。
例如 review 一定输出违规点、最小修改集、不可改区；refine 一定输出 patch strategy；generate 一定输出 shell strategy 和 non-goals。结构一旦稳定，才适合被 agent 化。

第三步，再引入 agent 的执行与反馈环。
此时 design-agent 才不仅能“给建议”，而是能在代码修改前后做审查、回归、偏差报告。

### 是否可能多 agent

是，而且很可能最终应该拆成至少两个角色：

- `generator`
  负责在规则约束下生成新页面方案或局部改造方案，强调“继承现有系统、低漂移”。

- `auditor`
  负责审查已有页面或 generator 的输出，重点识别：
  - 是否违反卡片边界
  - 是否出现渐变和阴影滥用
  - 是否破坏 header / 内容区主次
  - 是否偏离 light / dark 的统一语言

如果未来需要更细，可以继续拆成：

- `token-governor`：维护主题与语义 token 映射
- `preset-selector`：判断页面类型与最佳 preset

但在当前阶段，没有必要过早拆分。对小晴而言，先把 skill 层做稳，比过早引入多 agent 更重要。

## 6. 当前 v1 的架构选择

`Design Skill v1` 选择的是“轻结构、强约束”的方案：

- 用 Markdown 记录规则与提示模板，降低接入成本
- 用 YAML 记录 theme token 与 preset，方便未来被 agent 或脚本读取
- 文档只保留“架构”与“演进计划”两份，避免额外维护负担

这让它具备几个优势：

- 足够轻，能立刻放进当前项目使用
- 足够稳，能约束 AI 的常见过度设计问题
- 足够清晰，未来可逐步拆成更独立的 design-agent

结论上，`Design Skill v1` 不是一个“设计平台雏形”，而是一个“面向小晴当前 UI 系统的稳定设计约束层”。只有先把这层做对，未来的 design-agent 才不会失控。
