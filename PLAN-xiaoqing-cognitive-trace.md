# XiaoQing Cognitive Trace - Plan

> Transparent view of XiaoQing's cognitive evolution: how she perceives, remembers, decides, and adapts over time.

---

## Motivation

User life trace (PLAN-trace-points.md) captures **what happens in the user's life**. XiaoQing cognitive trace captures **what happens inside XiaoQing** -- her perception shifts, memory operations, decision adjustments, and personality evolution.

This builds trust through transparency ("I can see what she remembered and why") and makes XiaoQing's growth tangible instead of a vague feeling.

---

## Data Sources (already exist or partially exist)

| Source | Location | What it captures |
|--------|----------|-----------------|
| Debug Trace / TurnTraceEvent | `assistant/conversation/orchestration.types.ts` | Per-turn pipeline steps: intent, strategy, memory recall, tool calls, prompt assembly |
| Memory writes | `assistant/memory/` | When XiaoQing stores a new fact or preference about the user |
| UserClaim / CognitiveProfile | `assistant/memory/` (planned) | Pattern-level understanding: "user prefers plan-first", "user is stressed on weekdays" |
| Persona parameter changes | `assistant/persona/` | Style shifts: warmth, formality, humor levels |
| Reminder decisions | `assistant/conversation/` | When and why XiaoQing decides to nudge the user |
| Growth / identity events | `assistant/growth/` | Milestone achievements, capability expansions |

Key insight: **no new extraction pipeline is needed**. These events are already generated by existing modules -- we just need to surface and aggregate them.

## Phase 1 - MVP: Aggregate & Store Cognitive Events

**Goal**: A single `CognitiveEvent` table that collects notable XiaoQing-side events from existing modules.

### Tasks

- [ ] `CognitiveEvent` Prisma model
  ```
  id            String   @id @default(cuid())
  kind          String   // 'memory_write' | 'insight' | 'style_shift' | 'reminder_decision' | 'growth' | 'dialogue_adjustment'
  title         String   // Human-readable summary
  detail        String?  // Extended explanation
  source        String   // Which module produced it: 'memory', 'persona', 'orchestrator', etc.
  conversationId String? // Optional link to the conversation that triggered it
  confidence    Float    @default(1.0)
  happenedAt    DateTime @default(now())
  metadata      Json?    // Flexible payload per kind
  createdAt     DateTime @default(now())
  ```
- [ ] `CognitiveEventService` -- write + query + by-day grouping
- [ ] `CognitiveEventController` -- REST API (`GET /cognitive-events`, `GET /cognitive-events/by-day`)
- [ ] Emit events from existing modules:
  - [ ] Memory module: emit on `memoryStore.save()` / `memoryStore.update()`
  - [ ] Orchestrator: emit on notable strategy decisions (e.g., "chose to comfort before advising")
  - [ ] Persona: emit on parameter drift above threshold
- [ ] Backfill script for existing debug trace data (optional)

### Validation

> After 3 days of normal usage, the cognitive event stream should contain 5-15 events per day, covering at least 3 different kinds.

## Phase 4 - Cross-linking

**Goal**: Connect user life trace and XiaoQing cognitive trace bidirectionally.

### Concepts

- User TracePoint "I've been stressed about work" <-> XiaoQing CognitiveEvent "Noticed elevated stress pattern, switching to comfort-first responses"
- User plan "Going hiking this weekend" <-> XiaoQing reminder decision "Scheduled a gentle Friday evening hiking reminder"

### Tasks

- [ ] `relatedTracePointIds` field on CognitiveEvent
- [ ] `relatedCognitiveEventIds` field on TracePoint (or a join table)

## Architecture Notes

| Decision | Rationale |
|----------|-----------|
| Separate table from TracePoint | Different data owner (XiaoQing vs. user), different lifecycle, different query patterns |
| Event-based emission, not extraction | Cognitive events are system-generated, not LLM-extracted from conversation |
| No new LLM calls in Phase 1 | Just capture what existing modules already compute |
| Reuse existing UI components | Same timeline stream, badge, chip patterns from user life trace |
| `metadata: Json` for flexibility | Each kind has different payload shape; avoid over-engineering typed columns early |

---

## Risks

| Risk | Mitigation |
|------|------------|
| Too many low-value events | Filter by significance threshold; only emit when delta is meaningful |
| Performance of event emission | Async/fire-and-forget writes, no blocking on the main conversation path |
| User confusion between the two traces | Clear visual differentiation (different header, eyebrow label, color accent) |
| Privacy concern ("XiaoQing is watching me") | Frame as transparency, not surveillance; user controls visibility |
